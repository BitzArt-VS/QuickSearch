# Rendering Pipeline

`GuiDialog` rendering is driven through `GuiElementAdapter`, a minimal vanilla dialog registered with the GUI manager. Its `OnRenderGUI(deltaTime)` override calls `DialogRenderer.OnRenderFrame(deltaTime)`, so the custom renderer participates in vanilla dialog draw-order and focus stacking instead of being registered as a separate `IRenderer`.

**Invalidation model:** the renderer tracks three cascading dirty states:

1. Reconcile — scheduled by `RequestReconcile()`. Runs pending render fragments through `ScopedRebuildQueue`, then requests arrange.
2. Arrange — scheduled by `RequestArrange()` or by reconcile. Recomputes layout/input regions through the current combined tree walk, then requests paint.
3. Paint — scheduled by `RequestPaint()` / `RequestRender()` or by arrange. Redraws the Cairo surface from the latest arranged bounds and uploads the texture.

Arrange and paint are split in the dialog renderer. `ExecuteArrangeWalk` clears and re-registers input regions, recomputes layout, caches each slot's arranged bounds, paints the Cairo surface, and uploads the texture. `ExecutePaintWalk` resets paint-time tooltip regions, repaints from the cached arranged bounds through `GuiRenderTreeBuilder.Paint`, and uploads the texture without recomputing layout or input regions. Unchanged frames only blit the previously uploaded texture.

**Cairo surface:** physical-pixel-sized `ImageSurface` + `Context` allocated when the dialog renderer is created (`logicalW * GUIScale` rounded). Recreated on the fly when `RuntimeEnv.GUIScale` or the dialog's logical size changes. The CTM is scaled by `GUIScale` during paint so component code can work in logical pixels. On dirty paint, the surface is cleared (transparent), components draw into it, then `capi.Gui.LoadOrUpdateCairoTexture` uploads it to a `LoadedTexture` (reusing the GL texture when size is unchanged) and `capi.Render.Render2DTexturePremultipliedAlpha` blits it to screen at the texture's actual integer pixel size. Cairo's `Format.Argb32` uses premultiplied alpha, matching the blit call.

**Bounds:** `DialogRenderer` passes the full dialog surface bounds to the normal `GuiRenderTreeBuilder.Render` / `Paint` path. The renderer declares the dialog as a normal keyed slot, so the dialog is measured, positioned, rendered, overlaid, and registered for input like any other component. Standard dialogs set fixed root width/height through `ConfigureSlot`, matching the surface size; the renderer validates that both root dimensions are fixed after every dialog reconciliation. Descendants are arranged inside the dialog content bounds by the normal layout pass: measure → position → recurse. See [05.layout-parameters.md](05.layout-parameters.md) for full details.

**Measurement ownership:** Fit-content slot sizing resolves through `IGuiComponent.Measure`. The base `GuiComponent` default measures its own mounted child slots via `IGuiRenderHandle.Slot`, so the recursive measurement walk is component-owned. Raw fragments that need sizing are wrapped in a normal component slot instead of keeping a separate builder-owned child-measurement path.

**Floating layers:** `FloatingLayerRenderer` reconciles its active fragment inside an invisible root `GuiContainer` slot. Fixed-size overlays configure that root slot with the fixed size; auto-sized layers measure the root component with the placement max width/height. Tooltips therefore share the same component-owned recursive sizing semantics as normal dialog content.

**Dialog bootstrap:** `GuiDialogHostFeature` calls `GuiDialogHost.Initialize(clientApi)` during the client mod lifecycle and disposes `GuiDialogHost.Instance` during feature disposal. Feature code normally calls `GuiDialogHost.Toggle/Open<TDialog>(configure?)`; this prevents duplicate dialogs from separate feature-owned hosts. `new GuiDialogHostInstance(clientApi)` is available for explicit isolated ownership cases, and isolated hosts use instance methods such as `host.Open<TDialog>(configure?)`. Each host owns one typed `DialogRenderer<TDialog>` instance per dialog type, stored through the type-erased `DialogRenderer` base; `TDialog` is constrained to `class, IGuiDialog, new()` so `GuiDialog` is the default implementation, not the renderer's required base type. Renderer creation reconciles the dialog as a normal keyed slot and registers/opens the `GuiElementAdapter` (minimal vanilla `GuiDialog` that keeps `DialogsOpened > 0`, inherits vanilla dialog draw order, provides focus integration, and forwards input/render callbacks). Renderer removal closes/disposes the adapter and renderer; mounted dialog/tree slots are disposed only when their concrete nodes implement `IDisposable`. Calling `Open<TDialog>(configure)` with a non-null configure action for an already-open dialog re-runs dialog configuration/reconciliation on that existing typed renderer rather than creating a duplicate; calling `Open<TDialog>()` without configuration is an idempotent getter. Already-open reconfiguration is intentionally not transactional: if configuration, reconciliation, lifecycle hooks, child rebuild, or root-size validation throws, the exception propagates and the existing dialog may have been partially updated. Treat those failures as programmer errors, not as recoverable rollback points. `DialogRenderer<TDialog>` attaches an internal `GuiDialogRuntime` through `IGuiDialog.AttachDialogRuntime` during slot configuration so the dialog can request close/focus/cursor behavior without owning lifetime.

**Focus & input routing:** the adapter uses vanilla defaults — `ShouldReceiveMouseEvents` while opened, `ShouldReceiveKeyboardEvents` while focused. `GuiElementAdapter` forwards vanilla focus/mouse/key/Escape callbacks to `GuiInputRouter`; it no longer calls adapter-facing input methods on `DialogRenderer`. `GuiInputRouter` owns the full per-surface input path: recorded interactive/keyboard regions, mouse capture, hover transitions, root-slot key dispatch, focused-node key dispatch, host focus requests, surface/overlay containment checks, cursor priority, mouse-event handled rules, wheel focus gating, root-slot focus notifications, and the default Escape fallback. Dialog-wide keyboard hooks use the normal root component slot: concrete dialogs register root-slot key callbacks through `ConfigureSlot`, and the router dispatches those callbacks before focused-slot callbacks. Escape is framework-owned: the router blurs the focused component or closes the dialog. Mouse events are only marked `Handled` when they hit the dialog or overlay interaction surface — clicks outside propagate to other dialogs / HUD / world. Mouse-down on a hit also requests vanilla focus through a renderer-provided delegate. Mouse wheel is consumed only when the surface is focused AND hovered. The default `GuiDialog` base only registers the root focus callback needed to keep `IsFocused` current.

**Edge resize:** `IGuiResizable` exposes `SupportedResizeEdges` plus `Resize(GuiComponentBounds)`; it intentionally inherits only `IGuiMeasurable`, not `IGuiComponent`, so resize policy is separate from layout participation. During arrange, `GuiRenderTreeBuilder` automatically records a `ResizeRegion` for any arranged slot whose instance implements `IGuiResizable` and currently supports at least one edge. `GuiInputRouter` owns `GuiResizeController`, checks resize regions before normal mouse hit-testing, keeps the resize cursor active during gestures, and dispatches framework-suggested outer bounds to the resizable target. Normal component resize bounds are in surface-local coordinates; root-slot resize bounds are converted to screen-logical coordinates so dialogs can update their centre offset without knowing about the controller. `GuiDialog` implements `IGuiResizable`; `IsResizable` (default `false`) enables the **bottom edge, right edge, and SE corner by default** through the protected `ResizeEdges` hook, while public `SupportedResizeEdges` still returns `None` when resize is disabled. `MinWidth`/`MinHeight`/`MaxWidth`/`MaxHeight` (defaults 200/100/2000/1500) clamp the size in `GuiDialog.Resize`; they have no effect while `IsResizable` is false. Four custom cursors are registered with the platform layer on first arranged resizable region: `bitzart-uitw-resize-h`, `bitzart-uitw-resize-v`, `bitzart-uitw-resize-nwse`, and `bitzart-uitw-resize-nesw`. The PNG assets ship at `resources/assets/bitzartuitweaks/textures/gui/cursors/` alongside a `coords.json` declaring hot-points — the same layout vanilla uses at `assets/game/textures/gui/cursors/` (vanilla itself ships only `normal`/`textselect`/`linkselect`/`move`/`busy`, no resize sprites). Loading mirrors `ScreenManager.LoadCursor` exactly: `IAssetManager.TryGet` → `IAsset.ToBitmap(api)` → `ClientPlatformAbstract.LoadMouseCursor` (reached via reflection in `ClientApiExtensions.LoadMouseCursor`). macOS is short-circuited like vanilla. Setting `MouseOverCursor` on the adapter each frame applies the cursor — vanilla `GuiManager.OnRenderFrameGUI` reads that field unconditionally, so during an active resize gesture the cursor stays correct even when the pointer moves outside the dialog rectangle. Resize updates call `GuiDialog.Resize`, which mutates fixed width/height, derives `OffsetX`/`OffsetY` from the suggested screen bounds, and then calls `OnResizeUpdated(sizeChanged)`; the base implementation requests arrange when fixed size changed, while `ModConfigDialog` requests arrange on every update so arrange-time fractional child constraints are recalculated while dragging. The renderer's size/scale check still recreates the Cairo surface and requests the appropriate dirty state as a safety net.
